The Odero Mobile SDK emits structured diagnostic logs on each native platform to help you trace the payment flow (session loading, network calls, UI lifecycle and payment status) while integrating. This chapter explains how a third-party integrator views and captures these logs on iOS, Android and React Native.

Logs are emitted in DEBUG builds ONLY. Release builds are silent so that no diagnostic data — including anything that could resemble PII — is written to the device log. There is no runtime verbosity flag to toggle: log level is fixed and controlled entirely by the build configuration.

Log categories

On every platform the SDK groups its log lines into four functional categories so you can filter to just the area you care about:

CategoryWhat it covers
generalSDK lifecycle, initialization and high-level flow events.
networkOutbound requests to the Odero backend and their responses.
uiPayment sheet presentation, navigation and dismissal.
paymentPayment status transitions, 3DS handling and final results.

iOS

On iOS the SDK logs through Apple's unified logging system (os_log) under the subsystem com.odero.pay.sdk, with the four categories above (general, network, ui, payment).

Viewing logs in Console.app

  1. Connect your device (or use the Simulator) and open Console.app on your Mac.
  2. Select your device in the sidebar and press Start to begin streaming.
  3. In the search field, filter by subsystem: type subsystem:com.odero.pay.sdk.
  4. Optionally narrow further by category, e.g. category:payment.

Viewing logs from the terminal

You can stream the same logs from a terminal with the log command:

log stream --predicate 'subsystem == "com.odero.pay.sdk"'

To restrict the stream to a single category:

log stream --predicate 'subsystem == "com.odero.pay.sdk" && category == "payment"'

Capturing logs in-app

In addition to os_log, the SDK posts each log line as a NotificationCenter notification named "OderoLogger", with the formatted text under userInfo["message"]. Observe it to surface logs inside your own debug UI or forward them to your own logging pipeline:

NotificationCenter.default.addObserver(
    forName: Notification.Name("OderoLogger"),
    object: nil,
    queue: .main
) { notification in
    if let message = notification.userInfo?["message"] as? String {
        print("[Odero] \(message)")
    }
}

Verifying logging works

Call the helper below from a DEBUG build to emit one line of each level so you can confirm your filter or observer is wired up correctly:

OderoLogger.testLogging()

If you see no output: confirm you are running a DEBUG build (Release emits nothing) and that your Console.app / predicate filter matches the subsystem com.odero.pay.sdk exactly.

Android

On Android the SDK writes to Logcat using android.util.Log under four tags, one per category:

CategoryLogcat tag
generalOderoSDK-General
networkOderoSDK-Network
uiOderoSDK-UI
paymentOderoSDK-Payment

Viewing logs with adb

Filter Logcat to just the Odero tags:

adb logcat -s OderoSDK-Payment OderoSDK-Network OderoSDK-UI OderoSDK-General

Or grep across all output to catch every Odero line at once:

adb logcat | grep OderoSDK

Capturing logs in-app

To capture log lines inside your app (for example, to mirror them into your own crash reporter during development), set an optional context in Application.onCreate and register a callback. The callback receives the category type, the message and source metadata (file, function, line and timestamp):

class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()

        OderoLogger.appContext = this

        OderoLogger.logCallback = { type, msg, file, fn, line, date ->
            Log.d("MyApp-Odero", "[$type] $msg ($file:$line)")
        }
    }
}

Both OderoLogger.appContext and OderoLogger.logCallback are optional. Logcat output works without either; the callback only adds an in-process hook for DEBUG builds.

React Native

The React Native package bridges the native iOS and Android loggers. You can subscribe to log lines from JavaScript with addLogListener, which returns an EmitterSubscription. The listener receives an object with a single message field:

import { useEffect } from 'react';
import { addLogListener } from 'react-native-odero-mobilesdk';

useEffect(() => {
  const logSub = addLogListener((e) => {
    console.log('[Odero]', e.message);
  });

  return () => {
    logSub.remove();
  };
}, []);

Always call .remove() on the subscription in your cleanup function to avoid leaks and duplicate listeners across re-renders.

As on the native platforms, addLogListener only delivers events in DEBUG native builds. Release builds emit nothing. The underlying native logs remain viewable through the platform tools while developing:

  • Android: adb logcat | grep OderoSDK (or the tag-filtered form above).
  • iOS: Console.app filtered to subsystem:com.odero.pay.sdk, or the log stream predicate shown above.

Production behaviour

To summarize the privacy guarantee for going live:

  • Logging is active in DEBUG native builds only. Release builds produce no SDK log output on any platform.
  • There is no runtime flag to enable, raise or lower verbosity — behaviour is determined solely by the build configuration.
  • Because Release is silent, no diagnostic data or potential PII is written to device logs in production. You do not need to strip or guard any logging code before shipping.
