Test your integration end-to-end against Odero's DEV/sandbox environment before going live. Sandbox builds point at api-dev.pay.odero.ro and exercise the wallet (Apple Pay / Google Pay) TEST networks, so no real money moves and no production data is touched.

Never test with production credentials or production artifacts. Use the DEV/sandbox setup throughout development and QA, and switch to PROD only when you build a release. See Switching to production.

Enable the sandbox (DEV) environment

The environment is selected per platform at build time — there is no runtime toggle. Pick the DEV option that matches how you ship.

iOS

Add an sdk.plist to your app target with the ENV key set to DEV. This points the native iOS SDK at the DEV gateway and the Apple Pay TEST network.

<?xml version="1.0" encoding="UTF-8"?>
<plist version="1.0">
<dict>
    <key>ENV</key>
    <string>DEV</string>
</dict>
</plist>

Android

Depend on the -dev artifact. The DEV build resolves the Odero DEV gateway and runs Google Pay in TEST mode automatically.

implementation 'Odero.oderopaysdk:oderopaysdkandroid:1.0.0-dev'

React Native

Set the Oderomobilesdk_environment environment variable to DEV before/at build time. DEV pulls the -dev native artifacts (iOS DEV pod + Android -dev AAR); any other value builds for PROD. Rebuild the native app after changing it.

export Oderomobilesdk_environment=DEV

Get sandbox credentials and create DEV sessions

Request a set of sandbox merchant credentials (merchantId + merchantApiToken) from Odero. These are separate from your production credentials.

As in production, the payment session is created on your merchant backend (server-side) — never in the app. For sandbox, your backend calls the Odero DEV base URL https://api-dev.pay.odero.ro/ with your sandbox credentials, and returns only the resulting sessionId (a String) to the app. The app passes that sessionId to showPaymentSheet.

Backend step (not the app): Never embed merchantApiToken — sandbox or production — in the mobile app. Keep it server-side.

  1. Backend: POST util/merchant-signature-header-json with header X-Requested-With: XMLHttpRequest and body {merchantId, merchantApiToken, payload} → returns {merchant_signature}.
  2. Backend: POST api/payments/session with header X-MERCHANT-SIGNATURE → returns {sessionId}.
  3. Backend returns only the sessionId to the app; the app calls showPaymentSheet with it.

The amount, products, currency, customer/billing visibility, and Apple Pay / Google Pay visibility all come from the session you create on DEV — they are not passed to showPaymentSheet.

Direct card testing

For manual card entry in the sandbox payment sheet, use the standard sandbox PANs below. Use any future expiry in MM/YY format and any 3–4 digit CVV.

CardNumberResult
Visa4111 1111 1111 1111Approved
Mastercard5555 5555 5555 4444Approved
Declined card4000 0000 0000 0002Declined

These cards work in the sandbox/DEV environment only. The declined card is the easiest way to verify your failure handling (the failure status and errorMessage you receive from the payment event listener).

3DS challenge flows

3-D Secure (3DS) challenge flows are exercised in DEV. When a sandbox card triggers a challenge, the SDK presents the 3DS screen; complete it to confirm your app handles the success and failure outcomes correctly. On Android, if you intercept Activity results, remember to forward the SCREEN_3DS_REQUEST_CODE result to handle3DSScreenActivityResult(resultCode, intent).

Apple Pay sandbox testing

Apple Pay can only be tested on a real device (not the iOS Simulator), and the Apple Pay button appears only when the session enables it.

  1. Sign in to the device with an Apple sandbox tester account (created in App Store Connect → Users and Access → Sandbox).
  2. Add one of Apple's sandbox test cards to Wallet on that device.
  3. Pass your applePayMerchantId to showPaymentSheet and complete the flow.

Apple's sandbox tester accounts and test card numbers: https://developer.apple.com/apple-pay/sandbox-testing/

Make sure the In-App Payments entitlement lists your merchant id and that an Apple Pay Payment Processing certificate (gateway name oderopay) is attached to that Merchant ID, otherwise the button will not work even in sandbox.

Google Pay sandbox testing

With the -dev (TEST) artifact, the Android SDK runs Google Pay in the Google TEST environment automatically — gateway oderopay, networks Mastercard/Visa. The Google Pay button appears only when the session enables it and the device reports IsReadyToPay.

  1. Use a device or emulator that has Google Play services and is signed in to a Google account.
  2. Build with the -dev artifact so Google Pay resolves to TEST.
  3. Use cards from Google's test card suite to exercise the wallet flow.

Google Pay test card suite: https://developers.google.com/pay/api/android/guides/resources/test-card-suite

Verifying results during testing

Payment outcomes arrive asynchronously through the platform's event mechanism — there is no return value or promise from showPaymentSheet. While testing, confirm you receive each status (idle / processing / success / failure), the referenceNumber on success, and the errorMessage on failure.

  • iOS: observe NSNotification.Name.paymentStatusUpdated.
  • Android: set sdk.onPaymentStatusChanged.
  • React Native: register addPaymentEventListener (and remove it in cleanup).

Native DEBUG builds also emit SDK logs you can stream while testing — see the Logging chapter for how to view them on each platform.

Switching to production

When you are ready to release, swap every sandbox setting for its production counterpart and re-test with production credentials in a controlled rollout:

PlatformDEV / sandboxPROD / release
iOSsdk.plist ENV = DEVENV = PROD (or omit the key)
Android:1.0.0-dev artifact:1.0.0 release artifact
React NativeOderomobilesdk_environment=DEVany other value (PROD)
Backendsandbox credentials + https://api-dev.pay.odero.ro/production credentials + https://pay.odero.ro/

Do not ship the DEV artifacts, the ENV=DEV plist, the DEV env var, or sandbox credentials to production. They target api-dev.pay.odero.ro and the wallet TEST networks and will not process real payments.
