The Odero Pay Mobile SDK lets you accept payments inside your mobile app through a prebuilt, drop-in bottom-sheet checkout. You hand the SDK a payment session and it renders the entire pay UI, collects the payment method, runs the transaction, and reports the result back to your app.

The sheet handles raw card entry, Apple Pay, Google Pay, and 3-D Secure (3DS) bank authentication out of the box. The displayed amount, line items, currency, customer and billing fields, and whether the Apple Pay / Google Pay buttons appear are all driven by the session, so the same integration adapts per transaction without app changes.

What the SDK does

  • Card payments: a hosted card-entry form with a saved-cards list and CVV re-entry.
  • Apple Pay & Google Pay: native wallet buttons, shown only when the session enables the wallet (and, on Android, when the device is ready to pay).
  • 3-D Secure: the bank authentication step is presented automatically when the issuer requires it.
  • Prebuilt checkout UI: a themeable bottom sheet covering card details, CVV, processing, 3DS, and the success/failure result — no payment screens to build yourself.
  • Status callbacks: your app is notified of idle, processing, success (with a reference number), and failure (with an error message).

Distribution channels

The SDK ships through three channels. Pick the one that matches your app; the payment behavior and session contract are identical across all three.

Channel Module / package Distribution
Native iOS OderoPaySDKIOS CocoaPods pod from the Odero Artifactory podspec
Native Android com.oderopaysdk (Odero.oderopaysdk:oderopaysdkandroid) Gradle/Maven dependency from the Odero Artifactory repo
React Native (bridge) react-native-odero-mobilesdk npm package that bridges the native iOS + Android SDKs

The React Native package is a thin bridge over the two native SDKs, so the native iOS and Android requirements below also apply to a React Native app.

Prerequisites

Before integrating, make sure you have:

  • An Odero merchant account, which gives you a merchantId and a merchantApiToken.
  • Access to the Odero Artifactory repository to resolve the SDK artifacts (the iOS podspec source, the Android Maven repo, or both for React Native).
  • A merchant backend able to create payment sessions with Odero and return a sessionId to your app.

The shared session contract

All three channels follow the same model: the payment session is created on your merchant backend, never in the app. Using your merchantId and merchantApiToken, the backend asks Odero for a session and returns only the resulting sessionId (a string) to the app. The app then passes that sessionId to the SDK to show the sheet.

The amount, line items, currency, customer/billing visibility, and wallet (Apple Pay / Google Pay) availability all come from the session — they are not parameters you pass when showing the sheet. The amount is expressed in minor units (cents); the sheet displays it divided by 100.

Never ship your merchantApiToken in the app. It belongs only on the backend that mints sessions. The dev artifacts and dev configuration point at Odero's test environment and wallet TEST mode; do not ship them to production.

Getting started

  1. Provision an Odero merchant account and obtain your merchantId and merchantApiToken.
  2. Add a backend endpoint that creates a session and returns the sessionId.
  3. Install the SDK for your platform from the Odero Artifactory repo.
  4. Show the payment sheet with the sessionId and observe the payment status callbacks.
  5. Optionally theme the sheet and enable Apple Pay / Google Pay.

Continue with the platform-specific Quick Start guide for native iOS, native Android, or React Native.
