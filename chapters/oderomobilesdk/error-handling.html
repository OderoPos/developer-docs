The Odero Mobile SDK does not throw or return errors from showPaymentSheet. Every outcome — success, failure, and the in-between processing states — arrives asynchronously through a single result channel per platform. Your job is to subscribe to that channel, branch on a small set of statuses, and read referenceNumber on success or errorMessage on failure. This chapter documents that channel and gives a reference table of the errorMessage values you may encounter.

How results are delivered

There is exactly one place to listen on each platform. Register the listener before (or right when) you call showPaymentSheet, and keep it for the lifetime of the screen.

PlatformResult channelStatus values
React NativeaddPaymentEventListener(e => …)idle / processing / success / failure
iOS (native)NotificationCenter name .paymentStatusUpdatedpayment_idle / payment_processing / payment_success / payment_failure
Android (native)sdk.onPaymentStatusChanged = { … }payment_idle / payment_processing / payment_success / payment_failure

The payload always carries a status. On success it additionally carries a referenceNumber (the Odero transaction reference — store it, show it on your receipt). On failure it carries an errorMessage (one of the identifiers in the reference table below).

React Native — this is the only way results come back. There is no promise and no return value from showPaymentSheet.

import { showPaymentSheet, addPaymentEventListener } from 'react-native-odero-mobilesdk';

const sub = addPaymentEventListener((e) => {
  switch (e.status) {
    case 'processing':
      // show a spinner; do not navigate away
      break;
    case 'success':
      onPaid(e.referenceNumber); // reference for your receipt
      break;
    case 'failure':
      showError(mapError(e.errorMessage)); // map to your own copy
      break;
    case 'idle':
    default:
      break;
  }
});

showPaymentSheet(sessionId);

// always release on unmount
// sub.remove();

iOS (native) — observe the notification on the main queue.

NotificationCenter.default.addObserver(
  forName: .paymentStatusUpdated, object: nil, queue: .main
) { note in
  guard let status = note.userInfo?["status"] as? String else { return }
  switch status {
  case "payment_success":
    let ref = note.userInfo?["referenceNumber"] as? String
    // settle UI with ref
  case "payment_failure":
    let msg = note.userInfo?["errorMessage"] as? String
    // map msg to your copy
  default: break // payment_idle, payment_processing
  }
}

Android (native) — assign the callback before showing the sheet.

sdk.onPaymentStatusChanged = { status, referenceNumber, errorMessage ->
  when (status) {
    "payment_success" -> onPaid(referenceNumber)   // reference for your receipt
    "payment_failure" -> showError(mapError(errorMessage))
    else -> { } // payment_idle, payment_processing
  }
}

Cancellation is not a separate event

There is no dedicated "cancelled" status. When the user dismisses the sheet themselves, the SDK simply closes it — in most cases you receive nothing further, so treat the absence of a terminal result as "the user backed out" and leave your screen in its pre-payment state.

Some cancellations that happen mid-flow (during a wallet sheet or a 3DS challenge) instead surface as a failure whose errorMessage is a human-readable cancellation string, for example Payment cancelled by user, Google Pay cancelled, or 3DS authentication cancelled. Match these defensively and route them to a neutral "you didn't complete the payment" message rather than a hard error.

errorMessage reference

The value you receive in errorMessage (RN/Android) or userInfo["errorMessage"] (iOS) comes from one of two sources: a payment-status string returned by the gateway/issuer for a declined transaction, or an SDK error identifier for a problem the SDK itself detected (connectivity, session, 3DS, wallet). Map the ones you care about to your own localized copy; fall back to a generic message for the rest.

Do not branch on every one — map the common ones above and treat the rest with a generic decline message.

SDK error identifiers

These come from the SDK rather than the issuer. They typically point at configuration, connectivity, or a cancelled secondary flow. Each shows a canonical name and the raw string you may see on the wire.

Card-field validation errors (create_payment_card_expired, create_payment_card_number_invalid, create_payment_month_invalid, create_payment_year_invalid, create_payment_cvv_invalid) are handled inline by the payment sheet UI; you generally will not need to act on them.

Recommended approach

  • Branch only on the four statuses. Everything else is detail you read off the same event: referenceNumber on success, errorMessage on failure.
  • Maintain a small map from the known errorMessage values you care about to your own localized copy, with a single generic fallback for everything else. Do not try to enumerate the whole gateway list.
  • Treat ERROR_3DS_CLOSED and the *cancelled* strings as cancellations (neutral copy), not failures.
  • SESSION_NOT_FOUND almost always means a stale or wrong session — your recovery is to mint a fresh session on the backend, never to retry the same id.
  • When a message is opaque, read the detailed triplet (status, reference, message) from the SDK logs: tag OderoSDK-Payment on Android (adb logcat -s OderoSDK-Payment) and subsystem com.odero.pay.sdk on iOS (Console.app, or log stream --predicate 'subsystem == "com.odero.pay.sdk"'). Logs are emitted in DEBUG builds only.
