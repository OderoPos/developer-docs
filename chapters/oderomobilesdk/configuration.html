This chapter covers the configuration you need before showing the payment sheet: selecting the runtime environment, the merchant-backend session contract, and the wallet (Apple Pay / Google Pay) setup per platform.

Throughout this chapter, steps are clearly marked as either MERCHANT BACKEND (server-side, never in the app) or APP (in your mobile project).

Environment Selection

Every platform targets one of two environments. The DEV environment talks to api-dev.pay.odero.ro and runs wallets (Apple Pay / Google Pay) in TEST mode. The PROD environment talks to pay.odero.ro with live wallets.

Important: DEV artifacts, the DEV sdk.plist value, and the DEV environment variable all point at api-dev.pay.odero.ro and use wallet TEST mode. Never ship them to production.

iOS — sdk.plist ENV key

iOS selects the environment at runtime from a property list file. Add an sdk.plist file to your app target and set the ENV key:

ENV valueEnvironmentBase URLWallet
DEVDevelopmentapi-dev.pay.odero.roTEST
PROD (or key omitted)Productionpay.odero.roLive

sdk.plist — development

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>ENV</key>
    <string>DEV</string>
</dict>
</plist>

For production, set ENV to PROD or remove the key entirely.

Android — the -dev vs release artifact

Android has no runtime toggle. The environment is baked into the dependency you resolve: the -dev artifact is DEV (Google Pay TEST), and the plain release artifact is PROD.

app-level build.gradle — development (DEV / Google Pay TEST)

dependencies {
    implementation 'Odero.oderopaysdk:oderopaysdkandroid:1.0.0-dev'
}

app-level build.gradle — production

dependencies {
    implementation 'Odero.oderopaysdk:oderopaysdkandroid:1.0.0'
}

React Native — the Oderomobilesdk_environment env var

React Native chooses the environment at build time from the Oderomobilesdk_environment environment variable. Setting it to DEV selects the -dev native artifacts on both platforms; any other value (or unset) builds against PROD.

Set the variable before building the app

# Development build (api-dev.pay.odero.ro, wallet TEST)
export Oderomobilesdk_environment=DEV

# Production build — set to PROD or leave unset
export Oderomobilesdk_environment=PROD

You must rebuild the native app after changing this value — it is resolved during the native build, not at JS runtime.

Session-Creation Contract (Merchant Backend)

The payment session is always created on your merchant backend, never in the app. Your backend authenticates to Odero with your merchantId and merchantApiToken, creates a session, and returns only the resulting sessionId (a String) to the app. The app then passes that sessionId to showPaymentSheet.

Never ship merchantApiToken in the app. It is a server-side secret. The app receives nothing but the sessionId.

Odero base URLs

EnvironmentBase URL
DEVhttps://api-dev.pay.odero.ro/
PRODhttps://pay.odero.ro/

Step 1 (MERCHANT BACKEND) — obtain the merchant signature

Your backend signs the session payload by calling the signature-header endpoint. This is where merchantApiToken is used.

POST util/merchant-signature-header-json — header X-Requested-With: XMLHttpRequest

{
  "merchantId": "<your merchant id>",
  "merchantApiToken": "<your server-side api token>",
  "payload": { "...": "session request payload" }
}

The response contains the signature your backend forwards to the next call:

{ "merchant_signature": "<signature string>" }

Step 2 (MERCHANT BACKEND) — create the session

Your backend creates the session, passing the signature in the X-MERCHANT-SIGNATURE header.

POST api/payments/session — header X-MERCHANT-SIGNATURE: <merchant_signature>

{ "sessionId": "<session id string>" }

Your backend returns only this sessionId string to the app.

Amount and session-driven values

  • amount is in MINOR UNITS (cents). The payment sheet displays amount / 100.
  • amount, products, currency, customer / billing visibility, and wallet (Apple Pay / Google Pay) visibility all come from the session.
  • None of those are passed to showPaymentSheet — the app passes only the sessionId (plus optional appearance, saved cards, and on iOS the Apple Pay merchant id).

The app's only call is showPaymentSheet(sessionId, ...). Everything shown in the sheet is determined server-side by the session.

Apple Pay Setup (iOS & the RN iOS host)

Apple Pay setup is identical for native iOS and for the iOS host of a React Native app. The Apple Pay button appears in the sheet only if the session enables Apple Pay.

Configuration steps

  1. Create an Apple Pay Merchant ID in your Apple Developer account (for example merchant.token.applepay).
  2. Add the In-App Payments entitlement com.apple.developer.in-app-payments to your app target and list that exact Merchant ID.
  3. Generate and attach an Apple Pay Payment Processing certificate created for the Odero gateway, and attach it to your Merchant ID.
  4. Pass the same Merchant ID to the SDK as applePayMerchantId when showing the sheet.
  5. Test on a real device — Apple Pay is not available in the iOS Simulator.

com.apple.developer.in-app-payments entitlement — list your exact Merchant ID

<key>com.apple.developer.in-app-payments</key>
<array>
    <string>merchant.token.applepay</string>
</array>

Native iOS — pass the same id as applePayMerchantId

let sdk = OderoPaySDK()
// iOS has no parameterless init / .default(): pass the seven default page objects.
let appearance = OderoPaymentBottomSheetAppearance(
    oderoPaymentDetailsPageAppearance: OderoPaymentDetailsPageAppearance(),
    oderoPaymentProcessingPageAppearance: OderoPaymentProcessingPageAppearance(),
    oderoPaymentResultSuccessPageAppearance: OderoPaymentResultSuccessPageAppearance(),
    oderoPaymentResultFailurePageAppearance: OderoPaymentResultFailurePageAppearance(),
    oderoPaymentSummaryPageAppearance: OderoPaymentSummaryPageAppearance(),
    oderoPayWithAnotherCardPageAppearance: OderoPayWithAnotherCardPageAppearance(),
    oderoEnterCVCPageAppearance: OderoEnterCVCPageAppearance()
)
sdk.showPaymentSheet(
    sessionId: sessionId,
    appearance: appearance,
    applePayMerchantId: "merchant.token.applepay",
    savedCards: []
)

React Native — pass it through showPaymentSheet options

import { showPaymentSheet } from 'react-native-odero-mobilesdk';

showPaymentSheet(sessionId, {
  applePayMerchantId: 'merchant.token.applepay',
});

Apple Pay Payment Processing certificate

The Payment Processing certificate is generated against the Odero gateway (gateway name oderopay), not Apple's default processing. Generate the certificate for the Odero token platform and attach it to your Apple Pay Merchant ID so the token Odero environment routes and decrypts your Apple Pay transactions correctly.

These steps are required for the token Odero environment to respect your Merchant ID. The certificate is generated for Odero's oderopay gateway — do not use a generic / third-party processing certificate.

Google Pay Setup (Android & the RN Android host)

Google Pay is handled automatically by the native Android SDK. The gateway (oderopay), the merchant routing, the accepted networks (Mastercard / Visa), and the TEST-vs-PRODUCTION environment are all driven by the SDK plus the session. The Google Pay button appears in the sheet only if the session enables it and the device returns IsReadyToPay.

Configuration steps

  1. Add the Google Pay wallet dependency com.google.android.gms:play-services-wallet:20.0.0.
  2. Ensure the host Activity is a ComponentActivity (an AppCompatActivity or FragmentActivity satisfies this).

app-level build.gradle — add the wallet dependency

dependencies {
    implementation 'com.google.android.gms:play-services-wallet:20.0.0'
}

No JS parameter, manifest entry, or runtime gateway configuration is needed. The -dev artifact uses Google Pay TEST; the release artifact uses PRODUCTION. Routing to the oderopay gateway and the merchant id come from the session.

For React Native, the same two requirements apply to the Android host project — add play-services-wallet and make sure the host Activity is a ComponentActivity. There is no JS parameter for Google Pay.

Notes

  • The gateway / merchant routing and TEST/PROD selection are handled by the SDK and the session — you do not configure them in code or the manifest.
  • The button is shown only when the session enables Google Pay and Google Play services reports the device is ready to pay.
