The transaction-data command retrieves the bank response data for a previously executed payment transaction. By providing the transactionRequestId matching the requestId used in the original payment request, the integrator can obtain the outcome details returned by the bank for that transaction.

Important limitation: This endpoint returns data only for the last executed transaction. If the provided transactionRequestId does not correspond to the last transaction, the response will be equivalent to a failed transaction result.

Communication with the API for this command is achieved by sending a structured JSON object via a POST request to the following endpoint on the device:

http://{{deviceIP}}:9292/command/transaction-data

Usage flow:

  1. Client sends POST /command/payment with requestId: "rq_101"
  2. Server responds with the payment result for requestId: "rq_101"
  3. Client sends POST /command/transaction-data with transactionRequestId: "rq_101"
  4. Server responds with the bank transaction data
    POST: http://{{deviceIP}}:9292/command/transaction-data
{
  "requestId": "rq_102",
  "callBack": "",
  "transactionRequestId": "rq_101"
}

Request Parameters:

  • requestId (string, mandatory): A unique client-generated identifier for this request. Used to correlate the response with the request. Example: "rq_102", "txdata_001".
  • callBack (string, optional): A URL to which the server will send an asynchronous notification with the same response body after processing. If no callback is needed, this field can be null or an empty string ("").
  • transactionRequestId (string, mandatory): The requestId value that was used in the preceding /command/payment request. This identifies which transaction's bank data to retrieve. Example: "rq_101".

Response Body Structure:

The response will indicate the success or failure of the request, along with the bank data for the transaction.

    Successful response — transaction approved by bank:
{
    "requestId": "rq_102",
    "success": true,
    "extras": "{\"wasSuccessful\":true,\"bankResponseCode\":0,\"rrn\":null,\"approvalCode\":\"156\",\"message\":\"OK\"}"
}

    Successful response — transaction declined by bank:
{
    "requestId": "rq_102",
    "success": true,
    "extras": "{\"wasSuccessful\":false,\"bankResponseCode\":null,\"rrn\":null,\"approvalCode\":null,\"message\":null}"
}

    Error response — PGW app is not installed:
{
    "requestId": "rq_102",
    "success": false,
    "extras": "PGW app is not installed"
}

    Error response — no response from PGW:
{
    "requestId": "rq_102",
    "success": false,
    "extras": "No response from PGW"
}

Response Parameters:

  • requestId (string): The unique identifier that was sent in the request, used for correlation.
  • success (boolean): Indicates whether the HTTP request was processed successfully by the server.
    • true: The request was handled successfully. Note that a success: true response does not necessarily mean the bank approved the transaction — the bank result is contained within extras.
    • false: The request could not be processed due to a communication or internal error (e.g., PGW app not installed, no response from PGW).
  • extras (string): A JSON-serialized string containing the bank response data for the transaction. On error, this contains a human-readable error message.
    • wasSuccessful (boolean): true if the bank approved the transaction; false if it was declined or failed.
    • bankResponseCode (integer or null): The response code returned by the bank. 0 indicates success; 1 indicates failure. May be null if the transaction failed before reaching the bank.
    • rrn (string or null): Reference Retrieval Number — the bank's unique reference number for the transaction. May be null.
    • approvalCode (string or null): The approval code issued by the bank. Present only when the transaction was approved. May be null if declined.
    • message (string or null): A status message from the bank (e.g., "OK"). May be null if the transaction failed.

Timeout: This request has a default timeout of 60 seconds. If processing does not complete within this interval, the request will fail.
